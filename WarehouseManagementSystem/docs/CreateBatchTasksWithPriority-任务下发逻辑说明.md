# CreateBatchTasksWithPriority 任务下发逻辑说明

## 1. 方法定位

- 控制器方法：`MobileTaskController.CreateBatchTasksWithPriority`
- 文件位置：`Controllers/MobileTaskController.cs`
- 入口类型：`HTTP POST`
- 主要用途：从移动端库位页面批量创建用户任务，并在创建成功后锁定源位和目标位。

## 2. 前端入口

当前主要由以下页面触发：

- 页面：`Views/MobileLocation/Index.cshtml`
- 请求地址：`POST /MobileTask/CreateBatchTasksWithPriority`

前端请求体包含：

- `sourcePositions`：选中的源库位名称列表
- `targetGroup`：前端当前选择的目标组
- `priority`：用户选择的优先级，`1 = Low`，`2 = Medium`，`3 = High`

## 3. 当前固定路由

虽然前端会传入 `targetGroup`，但实际路由由后端再次确认：

- 源组是 `FG/PM`：只允许发往 `FG`
- 源组是 `RPM`：只允许发往 `FG/PM`

附加限制：

- `FG/PM -> FG` 只允许 `1A`、`2A`、`3A`、`1B`、`2B`、`3B` 作为起点
- `RPM -> FG/PM` 只允许 `FG/PM` 的 `4A`、`4B` 作为终点

## 4. 源位处理顺序

批量请求不会按前端点击顺序处理，而是先重排源位。

### 4.1 普通取货顺序

同一排 `A/B` 同时被选中时，源位处理顺序是：

```text
B1 -> B2 -> B3 -> ... -> A1 -> A2 -> A3 -> ...
```

也就是：

- 先按侧别排序，`B` 在前，`A` 在后
- 同侧内再按编号升序排序

这样做的目的，是满足“同排先取 B，再取 A”的现场规则。

### 4.2 为什么不是只看前端顺序

因为前端点击顺序不稳定，用户也可能是框选或乱序点击。后端统一重排，能保证任务行为稳定可预测。

## 5. 目标位分配顺序

### 5.1 `FG/PM -> FG`

目标 `FG` 的分配顺序是：

```text
A1 -> A2 -> A3 -> ... -> B1 -> B2 -> B3 -> ...
```

也就是：

- 先放完整个 `A` 区
- 再放 `B` 区

### 5.2 `RPM -> FG/PM`

目标 `FG/PM` 只检查 `4A`、`4B`。

排序仍按编号和侧别处理，但由于候选只有两个点，实际就是在 `4A`、`4B` 之间选可用终点。

## 6. 源位校验规则

每个源位在创建任务前都会检查：

1. 源位是否存在
2. 是否和目标组相同
3. 是否已有活动任务
4. 是否启用
5. 是否已锁定
6. 是否有物料
7. 是否被挡路

若属于可等待条件不满足，当前条目会先生成 `PendingCondition` 缓存任务，并在前端弹框中提示原因。后台 `PendingConditionTaskService` 会持续检查，等条件满足后自动释放为可执行任务。

以下属于可缓存等待的情况：

- A 位被同排 B 位挡路
- 源位未启用、锁定或暂时没有物料
- 目标区域暂时没有空闲工位
- 目标位在创建瞬间被其它活动任务占用

以下属于直接失败，不缓存：

- 源位不存在
- 源位已经有活动任务或已存在等待任务
- 源组和目标组相同
- `FG/PM` 起点不是允许发往 `FG` 的 `1A`、`2A`、`3A`、`1B`、`2B`、`3B`

## 7. 挡路判断规则

### 7.1 单独选择 A 位时

如果当前源位是 `A7` 这种 A 位，会检查同排的 `B7` 是否挡路。

以下情况会拦截：

- `B7` 已有活动任务
- `B7` 已锁定
- `B7` 未启用
- `B7` 有物料，不是空闲状态

提示文案示例：

```text
A7 cannot be picked because B7 is blocking the path.
```

如果整个 `B` 区都没有空闲位，则提示：

```text
由于B区挡住了去路无法拾取A。
```

### 7.2 同批次同时选择 A/B 时

这是当前逻辑里最重要的特殊处理。

如果 `A1` 和 `B1` 是同一次批量请求一起下发：

- 后端会先处理 `B1`
- 只有当 `B1` 的任务已经成功创建后，才允许后面的 `A1` 放行

这里不是单纯依赖数据库里的 `Lock` 字段，而是额外使用了一个本批次内存集合：

```csharp
createdSourcePositions
```

判断条件是：

1. 当前处理的是 `A` 位
2. 同一批次里存在同组、同编号的 `B` 位源点
3. 这个 `B` 位已经在当前请求中成功创建了任务

只有同时满足这三个条件，才会跳过“B 挡住 A”的拦截。

这表示：

- `A1` 单独下发时，仍然会被外部 `B1` 挡住
- `A1` 和 `B1` 同批下发，但 `B1` 创建失败时，`A1` 仍然会失败
- `A1` 和 `B1` 同批下发，且 `B1` 已成功建单时，`A1` 才允许继续创建

## 8. 同批次优先级叠加规则

用户在前端选择的是基础优先级。

后端会在此基础上再叠加一个“批次内顺序优先级”：

- `B` 位源任务：`实际优先级 = 用户选择优先级 + 1`
- `A` 位源任务：`实际优先级 = 用户选择优先级`

例如用户选择 `3`：

```text
B1/B2/B3 -> priority = 4
A1/A2/A3 -> priority = 3
```

这样可以确保底层严格按 `priority` 调度时，先执行 `B`，再执行 `A`。

## 9. 目标位可用性规则

候选目标位必须满足：

1. 在目标组内
2. 已启用
3. 未锁定
4. 没有物料
5. 没有活动任务
6. 本批次尚未被其它新任务占用

另外还有一条结构规则：

- 如果要选择某个 `A` 目标位，则同编号 `B` 目标位也必须是空闲状态

也就是说，目标 `A1` 可用的前提是 `B1` 也空闲。

如果找不到满足条件的终点，返回：

```text
该区域没有空闲状态的工位。
```

## 10. 成功后的处理

任务创建成功后，会立即执行：

1. 插入 `RCS_UserTasks`
2. 锁定源位 `RCS_Locations.Lock = 1`
3. 锁定目标位 `RCS_Locations.Lock = 1`
4. 将目标位加入本批次 `usedTargetLocations`
5. 将当前源位加入本批次 `createdSourcePositions`

其中：

- `usedTargetLocations` 用于防止同一批次重复分配同一个目标位
- `createdSourcePositions` 用于支持“同批次 B 已成功，A 允许继续”的规则

## 11. 缓存任务释放规则

缓存任务由 `PendingConditionTaskService` 处理，服务每隔约 2 秒检查一次 `taskStatus = PendingCondition` 的任务。

释放为正式任务前会再次检查：

1. 源位存在、启用、未锁定且有物料
2. 源位本身没有其它活动任务
3. A 位源点的同排 B 位已经让路
4. 路由仍然合法
5. 目标区域存在可用空位

释放成功后会：

1. 将任务状态从 `PendingCondition` 改为 `None`
2. 写入新的 `targetPosition`
3. 锁定源位和目标位

后台释放时也遵守当前排序规则：

- `PendingCondition` 检查顺序按 `priority DESC`，再按创建时间
- `FG` 目标位先选 A 区，再选 B 区
- `RPM -> FG/PM` 仍然只允许 `FG/PM` 的 `4A`、`4B`

## 12. 当前返回策略

接口采用“部分成功”策略：

- 只要整个方法没有抛出顶层异常，`success` 就返回 `true`
- 具体是否全部成功，要看 `createdCount` 和 `failedCount`
- `message` 会返回失败原因汇总，前端弹窗直接显示该原因

关键返回字段：

- `requestedCount`
- `createdCount`
- `failedCount`
- `pendingCount`：已经缓存等待条件满足的任务数

说明：

- `pendingCount > 0` 表示任务已经写入 `RCS_UserTasks`，状态是 `PendingCondition`
- 前端弹框会显示 `Queued Pending`
- 这些任务后续由 `PendingConditionTaskService` 自动释放

## 13. 典型示例

### 示例 1：`FG/PM` 同批选择 `A1,A2,A3,B1,B2,B3` 送去 `FG`

后端实际处理顺序：

```text
B1 -> B2 -> B3 -> A1 -> A2 -> A3
```

目标 `FG` 分配顺序：

```text
A1 -> A2 -> A3 -> B1 -> B2 -> B3
```

优先级示例，假设用户选 `2`：

```text
B1/B2/B3 = priority 3
A1/A2/A3 = priority 2
```

此时如果 `B1/B2/B3` 成功创建，那么后面的 `A1/A2/A3` 不会因为“刚刚被本批次锁住的 B”而失败。

### 示例 2：单独选择 `A7`

如果 `B7` 已被外部任务占用或本身不空闲，则直接失败：

```text
由于B7挡住了去路，无法拾取A7。
```

## 14. 维护注意点

后续如果要继续演进这套逻辑，最容易出问题的点有三个：

1. 不要把“本批次放行 A”误改成只看 `Lock`
2. 不要把源位排序改回 `编号优先`
3. 不要忘记 `B` 任务 priority 需要在用户选择值上再加一层

`PendingConditionTaskService` 必须和 `CreateBatchTasksWithPriority` 保持同一套路线规则，尤其是 A 位等待 B 位让路、`FG` 目标先 A 后 B、`RPM -> FG/PM` 只看 `4A/4B` 这三点。否则缓存任务释放时会绕过前端下发时的规则。
